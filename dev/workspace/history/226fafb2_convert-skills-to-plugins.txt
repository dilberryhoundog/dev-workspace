I've now read all five docs and mapped the current repo + the two skills as they run today in `dev-workspace`. Here's my assessment.

## How things work today (the constraints we're migrating)

- **The skills are self-contained.** Both `scripts/dev-workspace` and `scripts/dev-deploy` resolve their own location via `${BASH_SOURCE[0]}` and find `templates/`/`references/` relative to themselves. `conversation-capture` (Ruby) does the same with `__dir__`. ✅ This is the good news — the scripts will run fine from a plugin cache directory without changes.
- **The CLI must be on PATH in three places:** (a) your interactive terminal, (b) Claude's Bash tool, (c) the SessionStart/SessionEnd hooks. Today all three are satisfied by a stub in `~/.local/bin/dev-workspace` that walks up the tree to find a **project-local** `.claude/skills/dev-workspace/scripts/dev-workspace`.
- **That stub is the thing that breaks.** Once this is a plugin, the script lives in `~/.claude/plugins/cache/...`, not in the project. The walk-up stub will never find it → `❌ Not in a dev-workspace project`.
- **Hooks** (splash + `tree` on start, `tree` on end) live in the project's `.claude/settings.json` and call the bare `dev-workspace` command.
- **Preloaded context** (`.claude/rules/dev-workspace_commands.md`, `_guide.md`, `tree.md`, and the `@dev/workspace/CLAUDE.md` import) is project config. **Plugins do not load CLAUDE.md or rules** — only skills, agents, hooks. Most of that content already lives in `SKILL.md`, so this is mostly fine, but the auto-loaded guide won't auto-load anymore.

## Problems in the current `agent-library` scaffold

1. **Two parallel structures.** Real content is in `skills/`, and `plugins/*/skills/*` are symlinks back to `../../../skills/`. That's fragile and the sourcing is inconsistent (see #2).
2. **`git-subdir` + outward symlinks = broken install.** `dev-workspace` and `dev-deploy` marketplace entries use `git-subdir` with path `./plugins/dev-workspace`, but that dir's `skills/` is a symlink pointing **outside** the subdir (`../../../skills/`). Per the reference: *symlinks resolving outside the checkout are skipped for security*, and `git-subdir` only sparse-clones the subdir. **Those two plugins would install with empty skills.** Relative-path sources (whole-repo clone) dereference sibling symlinks fine; `git-subdir` does not.
3. **Mixed/duplicated sourcing.** dev-workspace & dev-deploy point at an *external* repo (`github.com/dilberryhoundog/dev-workspace.git`) while chat-tools uses local `./` with `strict:false`. This repo (`agent-library`) is itself the marketplace — pick one model. Also the source URL repo name (`dev-workspace`) doesn't match this dir (`agent-library`); need to confirm the actual remote.
4. **chat-tools is defined twice** — once via its own `plugin.json` and again in the marketplace entry (`strict:false` + explicit `skills` list). Redundant; let `plugin.json` be the authority and source `./plugins/chat-tools`.
5. **Version pinning conflicts.** chat-tools is `0.1.0` in `plugin.json` but `0.1.1` in the marketplace entry — `plugin.json` wins silently. And pinned versions mean pushing commits won't reach you unless you bump every time. For active dev, omitting `version` (commit-SHA) is simpler.

## What you've got right

## Real problems still open

**1. The launcher's `jq` doesn't match your registry shape — it will fail to resolve.**
Your `installed_plugins.json` stores each key as a **single object**, not an array, and there's **no `projectPath` field** (everything installs `user` scope):
```
"chat-tools@DBHG-Plugin-Marketplace": { "scope": "user", "installPath": "...", "version": "0.1.1", "gitCommitSha": "..." }
```
The launcher does `(.plugins[$key] // []) | map(...) | .[0].installPath` — `map`/`.[0]` assume an array, so this returns null against a bare object. It needs to be roughly `.plugins[$key].installPath`. The whole `projectPath`/cwd-preference block has nothing to match and should be dropped (or reworked if you ever use project scope).

**2. The launcher's `SCRIPT_REL` is wrong.** It says `scripts/dev-workspace`, but inside the plugin the script lives one level deeper under the skill: `skills/dev-workspace/scripts/dev-workspace`. Same for dev-deploy.

**3. Marketplace name mismatch.** Repo `marketplace.json` says `"name": "DBHD-Plugins"`, but the installed/known marketplace is **`DBHG-Plugin-Marketplace`** (note D**B**H**G**, not DBHD). The launcher's `PLUGIN_KEY="dev-workspace@your-marketplace"` is still a placeholder. These three names must agree or the launcher's registry lookup and `/plugin install` won't line up. Decide on the canonical name and set it everywhere.

**4. Hooks are NOT actually migrated — the `hooks/` dirs are empty.**
`plugins/dev-workspace/hooks/` and `plugins/dev-deploy/hooks/` exist but contain no `hooks.json`. The SessionStart (splash + `tree`) and SessionEnd (`tree`) hooks still only live in `dev-workspace/.claude/settings.json`. You'll need a `hooks/hooks.json` like:

  { "type": "command", "command": "\"${CLAUDE_PLUGIN_ROOT}\"/skills/dev-workspace/scripts/dev-workspace" },
  { "type": "command", "command": "\"${CLAUDE_PLUGIN_ROOT}\"/skills/dev-workspace/scripts/dev-workspace tree" }
]}]}}
```

"SessionStart": [ { "hooks": [ { "type": "command", "command": "dev-workspace session-start" } ] } ],
"SessionEnd":   [ { "hooks": [ { "type": "command", "command": "dev-workspace session-end" } ] } ]
```

That's it. In a non-dw project, `in_workspace` is false → `exit 0`, nothing prints, no error — on every session, everywhere. In a dw project, you get the splash + tree exactly as now. Manual `dev-workspace` (→ `info`) and `dev-workspace tree` are untouched, so manual UX doesn't change.

Why this shape over the alternatives:
- **Gating logic lives in the script, once** — the hooks become a single clean command each, and the "am I in a workspace?" rule isn't duplicated in JSON.
- Contrast with the quick hack `"command": "dev-workspace tree 2>/dev/null || true"`: that silences the tree *error*, but the bare `dev-workspace` splash still prints to stdout everywhere (redirect only catches stderr), and you'd be scattering shell guards across hook entries. The dedicated entrypoint suppresses both and reads cleanly.

Optional extra: `cmd_session_start` could still `exit 0` on a *failed* tree so a tree hiccup never noises up session start.
